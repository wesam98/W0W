### Objective

- Find where the lock pattern is stored in the AppLock app, understand how it's protected, and crack it to retrieve the original pattern.

---

### Steps

**Step 1: Check common storage locations**

- Started with `SharedPreferences` as it's the most common place apps store settings and credentials:
    
    ```bash
    adb shell
    cd /data/data/com.domobile.applock/shared_prefs
    cat *
    ```
    
- **Found this interesting entry:**
    
    ```xml
    <string name="image_lock_pattern">agYrmzRS42ZAcYGhv5Lqc+ntTEg=
    </string>
    ```
    
    - The pattern is stored under the key `image_lock_pattern`
    - The value `agYrmzRS42ZAcYGhv5Lqc+ntTEg=` looks like **Base64** — recognizable by the `=` padding at the end

---

**Step 2: Decompile and trace the code**

- Decompiled the APK using JADX and searched for `image_lock_pattern` to understand how the pattern is saved.
- **Found in JADX search results:**
    
    ```java
    gb.a(this.a, "image_lock_pattern", Base64.encodeToString(d(list), 0));
    ```
    
- This tells us:
    - The pattern goes through function `d(list)` first.
    - The result is then **Base64 encoded.**
    - Then stored in SharedPreferences under `image_lock_pattern` .

---

**Step 3 — Understand the `d()` function**

```java

static byte[] d(List list) {
    int size = list.size();
    byte[] bArr = new byte[size];

    // Step 1: convert pattern points to bytes
    for (int i = 0; i < size; i++) {
        f fVar = (f) list.get(i);
        bArr[i] = (byte) (fVar.b() + (fVar.a() * 3));
        // each dot = column + (row * 3)
        // dot 0 = top-left, dot 8 = bottom-right
    }

    // Step 2: hash the bytes with SHA-1
    return MessageDigest.getInstance("SHA-1").digest(bArr);
}

```

**What's happening step by step:**

- Each dot on the 3x3 grid is mapped to a number (0–8) based on its position:
| 0 | 1 | 2 |
| 3 | 4 | 5 |
| 6 | 7 | 8 |
- Those numbers are converted to bytes.
- The bytes are hashed with **SHA-1,** The **SHA-1 hash is Base64 encoded and stored**.
- The full protection chain:
Pattern dots → bytes → SHA-1 hash → Base64 encode → SharedPreferences.

---

**Why it's crackable**

- **SHA-1 is a one-way hash** — can't be reversed directly
- BUT the Android pattern only has **9 dots** and a very limited number of valid combinations
- **No salt is used** — same pattern always produces the same hash
- This makes it **brute-forceable** in under a second — try every possible pattern, hash it, compare to the stored hash

---

**Step 4: Crack the pattern**

<aside>
💡

Several tools exist online to crack Android pattern locks automatically. Two well-known ones are **APLCracker** (the tool used in this writeup) and **androidpatternlock**

</aside>

- Decode Base64 to get the raw SHA-1 hash bytes:
    
    `echo "agYrmzRS42ZAcYGhv5Lqc+ntTEg=" | base64 -d > gesture.key`
    
- **Run APLCracker to brute-force the pattern:**

```bash
python2 aplc.py gesture.key

```

<aside>
💡

A secure implementation would use a modern algorithm like **`bcrypt` or `Argon2**` with a random salt, making brute-force computationally infeasible even with the same small keyspace.

</aside>
